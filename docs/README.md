# 2차 프로젝트 — log·metric·trace 통합 장애 원인분석(RCA) 시스템

> **한 줄 요약**: log·metric·trace 3종 관측 지표를 통합 분석해 장애의 **원인(RCA)** 과 **전파 경로**를 자동 복원하는 시스템.

> 정본(正本) 문서는 Notion이며, 이 파일은 착수·진행 관리를 위한 통합 요약본입니다.

## 1. 배경 & 문제

- 기존은 **단건 로그** 위주로 장애를 판단 → "무엇이 문제인지"는 알아도 **어디서 시작해 어떻게 전파됐는지(인과·전파)** 는 파악 불가.
- log 데이터만으로는 안 잡히는 장애가 존재.
- 1차 프로젝트는 2005~2006년 BGL 데이터 기반의 도메인 특화 → 본 프로젝트는 **범용 장애 탐지·분석 시스템**으로 발전.
- **해결 방향**: observability 3요소(log·metric·trace)를 통합 분석 → log만으로 못 잡는 장애를 파악, 원인·전파 경로를 함께 복원.
- **차별점**: 단건 판정이 아니라 **여러 모달리티의 동시간대 정황을 근거**로 원인·전파를 복원.

## 2. 목표 & 범위

- **목표**: log·metric·trace 통합 분석으로 장애 원인(RCA)·전파 경로를 자동 복원.
- **범위**
  - ✅ SN 데이터 우선 검증 (우선순위 높음)
  - 🔜 TT 데이터 확장 검증 (우선순위 낮음)
  - ❌ 실서비스 자동 제어·복구·대응 → **사람(업무 담당자)의 역할, 시스템 범위 제외** (LLM agent 역할 경계 명시 필수)

## 3. 핵심 개념 — log / metric / trace

| 요소 | 정의 | 역할 |
|---|---|---|
| **Log** | 시점별 이벤트 기록(텍스트) | 무슨 일이 발생했는지 |
| **Metric** | 수치 시계열(CPU·지연·에러율 등) | 정상에서 얼마나 벗어났는지 |
| **Trace** | 요청이 서비스들을 거친 경로 | 어느 지점에서 장애 발생·전파됐는지 |

→ 세 요소를 **timestamp 기준 정렬·교차**해 원인 분석.

## 4. 시스템 아키텍처

```
저장소(AnoMod sample)
  → FastAPI/Python : 수집·정규화 → 트리거 감지 → 스냅샷 번들(정제) → 멀티에이전트 RCA
  → Spring        : 리포트 저장 + 조회/대시보드 API
  → React         : 리포트 조회 UI
```

- **컴포넌트 경계**: 데이터 계층(수집·트리거·필터·RCA) 전부 **FastAPI**에 집중. **Spring은 저장·조회만** 맡고 데이터를 운반하지 않음.
- 근거: 트리거를 Spring에 두면 원천 소비 wrapper가 중복 → 같은 데이터를 두 번 처리하게 됨. 프론트는 조회 용도이므로 Spring은 저장·서빙으로 충분.

## 5. 핵심 용어

| 용어 | 설명 |
|---|---|
| RCA | Root Cause Analysis, 장애 근본 원인 분석 |
| 모달리티 | log·metric·trace 각각의 데이터 유형 |
| Baseline | 정상 구간 기준으로 산출한 정상 프로파일 |
| 트리거 | baseline 대비 편차를 감지해 분석을 시작시키는 조건 |
| 스냅샷 번들 | 트리거 전후 데이터만 선별·정제해 LLM에 전달하는 정제 데이터 묶음 (원본 전체 X) |
| Detect-then-fetch | 트리거를 먼저 감지한 뒤 그 시점 전후 데이터만 fetch → 비용·성능 최적화 |
| SN / TT | AnoMod 데이터셋의 두 시스템 — SN(SocialNetwork), TT(TrainTicket) |

## 8. 대상 데이터 — AnoMod

- **출처**: EvoTestOps/AnoMod (Zenodo DOI **10.5281/zenodo.18342898**). TT·SN을 실제 구동해 장애를 주입·수집.
- **수집 모달리티**: 원본은 5종(log·metric·trace·api·templates)이나 **본 프로젝트는 log·metric·trace 3종만** 수집·분석.
- **장애 구성**: 두 시스템 모두 정상 1 + 장애 12종(성능 P·서비스 S·코드 C·DB D 각 3).
- **저장 축**: `data/{service}/{scenario}/{종류}`
  - 기본 3종 — `log_data` · `metric_data` · `trace_data`
  - 선택(확장) 2종 — `api_responses` · `code_coverage`
  - 원본 불변, 가공은 별도 경로.

### SN MVP 대상 시나리오 (normal은 baseline 산출용, 시나리오 수 미포함)

| 시나리오 | 장애 유형 | 주 트리거 신호 |
|---|---|---|
| `cpu_contention` | 성능(P) | metric CPU 임계 |
| `code_stop_media` | 코드(C) | 있어야 할 로그의 부재 |
| `svc_kill_media` | 서비스(S) | trace 에러 tag |

### 데이터 특성 & 조인

| 시스템 | Log Lines | Templates | Traces | Metrics | API Req | 역할 |
|---|---|---|---|---|---|---|
| SN (SocialNetwork) | 3,958.5K | 29 | 2,635 | 29 | 1,950 | 로그 중심 뼈대 |
| TT (TrainTicket) | 444.6K | 831 | 63,975 | 33 | 98,073 | trace 풍부한 멀티모달 깊이 |

- **SN**: log에 `req_id` 있음 → 로그만으로 요청 단위 묶기 가능. trace는 Jaeger 형식, 에러는 span의 tag(`http.status_code`·`error`)로 판단.
- **TT**: log에 `trace_id` 없음 → log↔trace는 **시각+서비스 근사 조인** (확장 시).
- **정규화 핵심**: 서비스 이름이 모달리티마다 다름 → **canonical 식별자 매핑**이 핵심.

## 9. 트리거 신호 (서비스 무관 공통 신호)

| 모달리티 | 신호 | 잡는 장애 | 우선순위 |
|---|---|---|:--:|
| log | 있어야 할 로그의 부재 | 서비스 정지(Code_Stop) | 상 |
| log | 평소 없던 로그(novelty) | 미지 오류 | 상 |
| log | start log 2회 확인 | 서비스 중단·재시작·예외(S·C) | 중 |
| metric | metric 임계(CPU·IO·net) | 성능(P) | 상 |
| metric | 응답 지연(latency 이탈) | DB 지연·성능(D·P) | 하 |
| trace | trace 에러 표시(is_error·tag) | 서비스 중단·재시작·예외(S·C) | 중 |

## 12. MVP 경계 & 일정

### MVP 성공 기준 (우선순위 M 전체)
1. SN 3개 시나리오(`cpu_contention`, `code_stop_media`, `svc_kill_media`) 대상
2. **[ 수집 → 정규화 → 트리거 → 스냅샷 번들 → 멀티에이전트 RCA → 리포트 저장 ]** 파이프라인 End-to-End 동작
3. 프론트에서 해당 RCA 리포트 조회

### MVP 검증 범위
- 각 시나리오 리포트가 **① 원인 서비스 ② 장애 유형**(시나리오명으로 식별)을 맞히면 통과.
- **3개 시나리오 3/3 통과**가 목표.

### 확장 범위
- TT 시나리오 추가 검증·보완
- 일반화 검증(FR-V-03), 시각화(FR-S-05), 검증 에이전트(FR-A-05)

### 🎯 발표 실증 목표 (7/29)
- TT 최소 1~2 시나리오까지 검증 → **일반화(서비스 무관 RCA)를 데이터로 실증**.
- 일정 미달 시: SN 3종 검증 + 일반화 설계(서비스 무관 편차·공용 파이프라인) 설명으로 갈음.
