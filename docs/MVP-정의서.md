## 1. MVP 한 문장 정의

> **SN 3개 장애 시나리오에서 [ 수집 → 정규화 → 트리거 → 스냅샷 번들 → 전송(/ingest) → 멀티에이전트 RCA → 리포트 저장 ] 파이프라인이 End-to-End로 동작하고, 프론트에서 각 RCA 리포트를 조회할 수 있으면 MVP 성공.** (7/7 회의 — 기획서 §7 기준)

리포트가 원인 서비스·장애 유형을 맞히는지(정확도)는 MVP 성공 기준이 아니라 **별도 검증 단계(Gate3 채점)의 지표**다.

## 2. MVP 목표 (Definition of Done)

MVP는 아래 2가지가 모두 충족되면 완료다 (D-016).

1. **파이프라인 E2E 동작** — SN 3개 시나리오에 대해 7단계 파이프라인(전송 포함)이 사람 개입 없이 끝까지 돌아 리포트가 생성된다.
2. **리포트 조회** — 프론트(React)에서 생성된 RCA 리포트를 조회할 수 있다.

**검증 목표 (MVP 성공 기준 아님, Gate3 채점)**: 3개 시나리오 리포트가 각각 ① 원인 서비스 ② 장애 유형을 맞힌다 → **3/3 통과**가 목표. 정답지는 **원본 시나리오 명칭** 기준. §5 검증 기준표로 판정.

## 3. 아키텍처 경계 (엣지 / 중앙)

- **엣지 수집기 (순수 Python SDK)**: 수집·정규화·baseline·트리거·선별·번들 구성을 소스 가까이에서 **로컬 처리**. 트리거 발화 시에만 스냅샷 번들을 중앙으로 전송(detect-then-fetch). **SDK 실배포는 제외, 로컬 패키징까지가 범위.**
- **중앙 FastAPI**: 수집 API(`POST /ingest`, 단일 진입점) + 멀티에이전트 RCA.
- **Spring**: 리포트 저장·조회만 (데이터를 운반하지 않음, D-001). **React**: 리포트 조회 UI.

## 5. 대상 시나리오 & 검증 기준 (Gate3 채점용 — MVP 성공 기준 아님)

`normal`은 baseline 산출용 (시나리오 카운트 미포함). 결함군·트리거 신호는 [고유경 Perf 에러 데이터분석](https://app.notion.com/p/3961bb95d18e808a948fc16133c22a0c) 결함군 대비 기준.

| # | 시나리오 | 결함군 | 기대 원인 | 기대 장애 유형 | 주 트리거 신호 | 판정 |
|:--:|---|---|---|---|---|:--:|
| 1 | `cpu_contention` | 지연 전파형 | **호스트(노드) 레벨 CPU 경합** — 특정 컨테이너 아님 (실측: 컨테이너 CPU 고른 분포 + 호스트 100%) | 성능(P) | metric CPU **임계(50%) 초과 빈도·지속시간** (평균/중앙값으론 안 잡힘 — 실측 median 3.89% vs baseline 3.58%) | ☐ |
| 2 | `code_stop_media` | 실패 전파형 | media 서비스 | 코드(C) | 있어야 할 로그의 부재 + **trace 500·NginxThrift 에러** | ☐ |
| 3 | `svc_kill_media` | 다운타임형 | media 서비스 | 서비스(S) | **로그 침묵(104s) + start log 2회** (재시작 마커 — Q-005 실측, Perf 분석 결함군 대비와 일치) · ~~trace 에러 tag~~(실데이터에 error tag 0건) | ☐ |

> **판정 규칙**: 리포트가 해당 행의 **원인 + 장애 유형**을 모두 맞히면 그 시나리오 통과(☑). 정답지는 원본 시나리오 명칭. media 계열 원인 서비스의 정확한 이름은 데이터 라벨 확인 후 확정(Q-003).
> **오염 신호 사용 금지**: API 성공률·상태분포, SocialGraph 로그 볼륨은 baseline과 동일한 워크로드 아티팩트 — 트리거·판정에 쓰지 않는다 (Perf 분석 §6).
> **정답 유출 금지 (유스케이스·기능정의서 v0.2 원칙)**: LLM 입력에서 시나리오 명칭·정답 라벨·파일명 제거 — 트리거=이상 감지 / LLM=원인 판단 분리. `/ingest`의 `title`·`bundle_id`(시나리오 명칭 포함)는 저장·화면 표시용, **LLM 분석 입력에서 제외** (api-spec §3.1).

**시나리오별 세부 조건**

MVP 통과 조건 (§2 DoD):
- [ ] 트리거가 baseline 편차로 발화 (시나리오 정답에 의존하지 않고 감지됨)
- [ ] 스냅샷 번들이 원천 전량이 아닌 ±Δ 구간으로 구성됨

- [ ] RCA 리포트 생성됨
- [ ] 프론트에서 해당 리포트 조회 가능

검증 통과 조건 (Gate3, VAL-01):
- [ ] RCA 리포트가 원인(서비스/호스트 레벨) 지목

## 7. 사용자·화면·예외 (최소 명세)

### 7.1 사용자 스토리

- **리포트 조회자**는, 장애의 원인·전파 경로를 빠르게 파악하기 위해, 리포트 목록에서 시나리오별 상세 리포트를 조회한다.
- **엣지 수집기**와 **분석 백엔드**는(구 "분석 시스템" 통칭 — v0.2에서 액터 분리), 사람 개입 없이 RCA 리포트를 만들기 위해, 트리거 발화 시 파이프라인을 자동 실행한다.

### 7.2 UI/UX 최소 요구

| 화면 | 필수 요소 |
|---|---|
| 대시보드 | KPI 카드 3종(전체 리포트·HIGH·오늘 발생) · 최근 리포트 5건 · 상세 이동 (`/api/dashboard`, api-spec §4.3) |
| 리포트 목록 | 심각도 · 시나리오명(장애 유형) · 생성 시각 · 원인 서비스 요약 · 상세 이동 |
| 리포트 상세 | 원인 서비스 · 장애 유형 · 전파 경로 · 근거(스냅샷 발췌) · 대응 방안 |
| 공통 | 데이터 없음 / 로딩 / 오류 상태 메시지 표시 |

### 7.3 입력값·예외 처리

| 상황 | 대응 |
|---|---|
| 트리거 미발화 (편차 없음) | 분석 미실행 — 정상 동작으로 간주, 로그만 남김 |
| 스냅샷 번들 공백 (±Δ 구간에 데이터 없음) | RCA 실행 중단, 리포트 대신 "근거 부족" 상태 기록 |
| `/ingest` 전송 실패 | 엣지에서 제한 횟수 재시도, 초과 시 로컬 보존 + 실패 로그 |
| LLM 응답 실패·형식 오류 | 제한 횟수 내 재시도, 초과 시 실패 리포트 저장 (내부 기록용 — 프론트는 DONE만 조회, api-spec v0.2.1) |
| 조회 시 리포트 없음 | 빈 목록 + 안내 메시지 (오류 아님) |

### 7.4 권한

- 내부 팀 도구 — MVP에서는 **인증 없음**(비로그인 조회 허용, D-013). 외부 공개 시 재검토.
