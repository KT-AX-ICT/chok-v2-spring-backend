# 🛠️ Git Convention

> 출처: 노션 "Git Convention" (팀스페이스 홈 › 한눈에 보는 작업 명세서)
> 정리: 2026-06-19
> 협업 시 코드 변경 이력을 쉽게 파악하고 관리하기 위한 **AngularJS 커밋 메시지 규칙** 적용.
> 커밋 메시지는 `제목(type)`, `본문(body)`, `꼬리말(footer)`로 구성하며 **빈 행으로 구분**한다.

---

## 📌 1. 커밋 타입 (Type)

커밋의 목적을 나타내는 태그. 주로 영어 **소문자**로 작성.

| Type | 설명 |
|---|---|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `docs` | 문서 수정 (README 등) |
| `style` | 코드 포맷팅, 세미콜론 누락 등 (코드 변경 없음) |
| `refactor` | 코드 리팩토링 (기능 추가/수정 제외) |
| `test` | 테스트 코드 추가/수정 |
| `chore` | 빌드 업무 수정, 패키지 매니저 설정 등 |

---

## 📝 2. 커밋 제목 (Subject)

변경 사항에 대한 요약으로 **50자 이내**로 작성.

- **명령문 사용**: 과거형('수정함') 대신 명령형('수정')으로 작성
- **마침표 생략**: 문장 끝에 마침표(`.`)나 느낌표(`!`)를 붙이지 않음
- **예시**: `feat: 로그인 화면 UI 추가`

---

## 📖 3. 커밋 본문 (Body)

선택 사항이지만, 무엇을 왜 수정했는지 상세히 작성할 때 사용.

- 제목과 빈 줄을 하나 둔 후 시작
- 어떻게(How)보다는 **무엇(What)과 왜(Why)** 를 작성

---

## 🏷️ 4. 꼬리말 (Footer)

선택 사항으로, 이슈 트래커 ID나 참고 정보 등을 작성할 때 사용.

- **예시**: `Close #123` 또는 `Resolves: #456`

---

## 💡 표준 및 심화 가이드

더 자세한 규격과 자동화 도구에 대한 정보는 다음 문서를 참고.

## ✍️ 커밋 메시지 예시

```
feat: 분석 완료 로그 목록 조회 API 추가

분석 상세 화면에서 log_analysis row가 존재하는 로그만
페이지네이션으로 조회할 수 있도록 GET /analysis 엔드포인트 신설.

Resolves: #12
```

> 브랜치 네이밍 규칙은 미정 상태이며, 회의록("06-18 개발 관련 회의")에
> "프레임워크별 컨벤션 추가 논의 필요, Git 컨벤션도 필요"로 추가 논의 예정 표시됨.
> 확정 전까지 이 저장소에서는 `feature/<작업명>`, `fix/<작업명>`, `chore/<작업명>` 형태를 임시 사용.
